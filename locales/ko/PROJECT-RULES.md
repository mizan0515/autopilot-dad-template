## 진실의 출처

문서 충돌 시 우선순위:
1. `{{PRD_PATH}}` — 제품 요구사항 / 설계 의도
2. 운영자가 `.autopilot/config.json` 에 선언한 상위 문서
3. 이 저장소 내부의 해당 기능 명세
4. 인라인 주석과 커밋 메시지

하위 문서는 상위 문서의 핵심 수치, 규칙 소유권, 표준 용어를 재정의하면 안 된다. 하위 요약이 낡았다면 상위 문서를 따르고, 가능하면 같은 작업에서 낡은 요약을 갱신한다.

## 저장소 현실 확인

낡은 요약, 채팅 로그, 임시 계획을 신뢰하기 전에 이 섹션을 먼저 읽는다.

- 서술형 요약보다 실제 파일 인벤토리를 우선한다. 파일 존재를 주장하기 전에 `rg --files <경로>` 또는 `git ls-files <경로>` 로 확인한다.
- 파일이 이미 존재하는데도 "미구현"으로 설명하는 낡은 요약이 흔하다. 발견하면 같은 작업에서 갱신한다.
- 문서나 로그에서 언급된 파일 이름에 의존하기 전에 존재를 확인한다.

## 에이전트 가드레일

- 공통 파이프라인이 있는데 일회용 로직으로 우회하지 않는다.
- 도메인이 허용하면 데이터 기반으로 유지. 공통 리졸버로 표현할 수 있을 때 케이스별로 로직을 포크하지 않는다.
- 핵심 규칙을 전송 콜백, UI 스크립트, 일회용 글루 코드 안에 묻지 않는다.
- 문서, 채팅 노트, 실제 파일이 불일치하면 추측하지 않는다. 실제 파일을 확인하고, 낡은 안내를 갱신한다.
- **검색 루트**: LLM 검색은 운영자가 `.autopilot/config.json` → `search_roots` 에 선언한 소스 디렉터리로 한정한다. 캐시 디렉터리(`Library/`, `Temp/`, `node_modules/`, `target/`, `build/`, `dist/`, `.autopilot/.archive/`, `Document/.archive/`) 와일드카드 검색 금지.
- **아카이브 건너뛰기**: `.archive/` 서브트리는 역사적 자료. 먼저 아카이브 `INDEX.md` 한 줄 요약을 읽고, 꼭 필요할 때만 특정 아카이브 파일을 연다.
- Git 시간 쿼리: 상대 날짜(`"1 week ago"`) 대신 커밋 해시 또는 절대 날짜(`--since=2026-04-01`)를 쓴다.

## 구현 전략

- 광범위한 프레임워크 작업보다 수직 슬라이스를 우선한다.
- 일반적인 작업 크기: 1-3 소스 파일, 1 데이터 스키마, 1 UI 화면, 1 모듈 통합, 1 엔드-투-엔드 플로우 중 하나.
- 코드 경로, 데이터 연결, 최소 검증이 모두 존재해야 작업 완료다.
- 데이터 표면이 큰 시스템에서는 검증과 디버그 가시성이 작업의 일부지, 추후 보완 항목이 아니다.

## 문서 갱신 규칙

- 프로젝트가 메타데이터 블록을 쓴다면, 추적되는 모든 문서에 보존한다.
- 인코딩: 에이전트 대상 Markdown (`AGENTS.md`, `CLAUDE.md`, `DIALOGUE-PROTOCOL.md`, `PROJECT-RULES.md`, `.claude/commands/**/*.md`, `.prompts/**/*.md`) 은 UTF-8 with BOM 을 유지한다.
  - **예외:** Codex 스킬 런타임 파일 `.agents/skills/**/SKILL.md` 와 `.agents/skills/**/agents/openai.yaml` 는 UTF-8 without BOM 이어야 한다 (Codex 가 0바이트부터 frontmatter/YAML 을 로드).
  - 비-ASCII 텍스트가 깨져 보이면 진행을 멈추고 인코딩을 먼저 고친다.
- 의미 있는 문서 작업 후 `tools/Validate-Documents.ps1` 을 돌린다. BOM 정규화에 `-Fix`, 에이전트/프로토콜 Markdown 을 건드리면 `-IncludeRootGuides -IncludeAgentDocs`.
- 표준 수치나 용어를 바꾸면 영향받는 모든 문서를 같은 패스에서 갱신한다.
- 문서, 코드 주석, 로그, 테스트 이름 간 용어를 일관되게 유지한다.

## 리서치 파일 규칙 (선택)

프로젝트가 폴더별 리서치 파일을 쓴다면, 해당 폴더에 파일을 추가/삭제/수정할 때마다 같은 작업에서 리서치 파일을 갱신한다. 후속 작업으로 미루지 않는다.

리서치 갱신에 담을 것: 신규 파일과 목적, 삭제된 파일과 이유, 동작 변경, 새로 생긴 갭, 해소된 갭.

## 테스트와 검증 기준

- 프로덕션 코드 변경 후에는 변경 범위에 맞는 가장 좁은 검증 단계를 실행한다.
- 가능하면 프로젝트 네이티브 검증 플로우 (테스트 러너, 린터, 타입 체커, 통합 하네스) 를 우선한다.
- 전체 검증이 무관한 이슈로 막히면 가장 좁은 로컬 체크만이라도 실행하고 막힌 원인을 명확히 기록한다.
- "될 거야", "괜찮아 보인다" 같은 표현을 검증된 사실의 대체물로 쓰지 않는다.

## 편집과 셸 폴백 규칙

- 코드 편집은 구조화된 patch 도구를 우선한다. Windows 에서 실패하면 명시적 인코딩 PowerShell 쓰기로 폴백 (`[System.IO.File]::ReadAllText/WriteAllText` + `UTF8Encoding $false` 로 BOM 없음, 또는 BOM 포함 생성자).
- 인코딩 민감 파일 (현지화 문자열, 비-ASCII) 을 편집하기 전에 정확한 타깃 라인을 먼저 읽는다. 편집 후 쓰기가 안전하다고 가정하지 말고 편집 주변을 즉시 다시 읽는다.
- 현지화 파일에서 문자열 하나가 깨졌다면, 마지막으로 알려진 정상 소스(`git show`, 표준 스펙)로 확인할 때까지 주변 문자열도 안전하지 않다고 가정한다.
- 현지화 문자열 복구 작업에서는, 교체 표현을 창작하기 전에 마지막 정상 소스에서 텍스트를 복원한다. 재작성이 아니라 복구로 취급한다.
- 셸 폴백 쓰기 시에는 기존 인코딩과 개행 스타일을 보존한다.
- 문자열, 보간 문자열, 라벨을 건드린 편집 후에는 성공 선언 전에 구문 중심 가드 체크 한 개 이상을 실행한다: 따옴표 밸런스 스캔, `git diff --check`, 사용 가능한 가장 좁은 컴파일/테스트 검증.

## 대화 로깅

- 작업 세션 요약을 `Document/chat/` 에 `YYYY-MM-DD-HHMM-topic.md` 형식으로 남긴다.
- 기록할 것: 요청, 결정, 변경 파일, 차단 요인, 다음 단계. 원본 대화는 저장하지 않는다.

## Git 워크플로

- 의미 있는 갱신마다 커밋을 만든다. 자체 완결되고 검증된 후에만 푸시한다.
- 워크트리가 무관한 이유로 더러우면, 커밋/푸시를 건너뛴 사실과 이유를 보고한다.
- 커밋/푸시를 조용히 건너뛰지 않는다. 최종 보고는 커밋/푸시했거나 왜 하지 않았는지를 명시한다.
- `--no-verify` 를 쓰지 않는다. Pre-commit 훅이 템플릿 불변식을 강제하며, 우회는 템플릿 계약을 훼손한다.
- 브랜치 정책: 현재가 `main`/`master` 이면 새 작업 브랜치를 만든다. main 에 직접 푸시하지 않는다.
